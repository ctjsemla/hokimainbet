## The Trust Problem in Traditional Casinos

Before Provably Fair makes sense, you need to see why trust became such a persistent issue in the gambling industry.

### The Black Box Players Cannot Audit

At conventional online and land-based casinos, results come from a Random Number Generator running on the operator's servers. Players never see the source code, cannot inspect the seeds in use, and have no independent way to confirm that outcomes were not altered after a bet was placed.

Operators display certificates from testing labs such as eCOGRA or iTech Labs. Those certificates are reassuring, but they remain a **second layer of trust**: you trust the auditor, the auditor trusts the operator, and the operator controls the entire infrastructure.

### Manipulation Risk Is Real, Even If Rare

RNG manipulation scandals have appeared at many scales — from small operators quietly changing payout ratios to larger cases involving software providers. Licensed major brands reduce risk, but the probability is never zero. Players who lose heavily often feel cheated, and without mathematical proof, that feeling is hard to refute or confirm.

## What Is Provably Fair?

Provably Fair is a cryptographic protocol that lets players **mathematically prove** that game outcomes were determined before bets were placed and could not be changed afterward by the operator.

The word "provably" is not marketing fluff. It refers to properties you can check with public tools, without server access.

### Basic Principles

The system rests on three main components:

1. **Server seed** — a random string generated by the casino before play begins.
2. **Client seed** — a string controlled or randomized by the player.
3. **Nonce** — a counter that increments each round or bet.

These elements combine through a hash function (often SHA-256 or HMAC-SHA256) to produce random numbers that drive outcomes: Crash multipliers, Dice rolls, Mines positions, Plinko ball paths, Wheel segments, Keno draws, and more.

### Why the Hash Matters

Before play starts, the server publishes the **hash of the server seed**, not the seed itself. Hashing is one-way: you cannot reverse a hash to recover the original seed. After the seed is revealed (rotated), anyone can hash it again and confirm it matches the pre-game hash.

That prevents a scenario where the server sees your bet, then picks a seed that produces a losing outcome. The seed is "locked" via hash before your wager lands.

## Client Seed, Server Seed, and Nonce — Deep Dive

Understanding each piece makes verification straightforward instead of intimidating.

### Server Seed

The server seed is a long random string generated by the casino system. Example shape: `a7f3b2c9d1e8f4a6b5c3d2e1f0a9b8c7`. It stays on the server and is **not shared** while active.

What players receive is the **hash** of that seed — often a SHA-256 string like `3f2a1b9c8d7e6f5a4b3c2d1e0f9a8b7c6d5e4f3a2b1c0d9e8f7a6b5c4d3e2f1`.

Save that hash before you bet. When the server seed rotates (on request or after a set number of rounds), the plaintext seed is revealed. Re-hash it. If the result matches your saved hash, the server could not have swapped seeds retroactively.

### Client Seed

The client seed is a string **players can influence**. On many platforms you can:

- Set a custom client seed manually.
- Let the system generate one randomly.
- Change it anytime — which often triggers server seed rotation as well.

Client seed gives players agency. Without it, a server could theoretically choose seeds that hurt everyone. With unique client seeds per player, mass manipulation becomes far harder.

### Nonce

Nonce means "number used once" in practice here: a counter that increments every bet or round. Round one might use nonce 0, round two nonce 1, and so on.

Nonce ensures different outcomes even when server and client seeds stay the same. Without it, identical seed pairs would repeat identical results — useless for multi-round games.

### Typical Verification Formula

Implementations vary by platform, but the pattern is consistent:

```
result = HMAC_SHA256(server_seed, client_seed + ":" + nonce)
outcome = derive_game_result(result)
```

The `derive_game_result` function differs per game: Crash converts the hash to a crash multiplier, Dice to a number from 0 to 100, Mines to mine positions, Plinko to a left/right path sequence. Transparent platforms publish this logic in help docs or Provably Fair pages.

## Verification Steps: A Practical Walkthrough

Verifying Provably Fair is not a programmer-only ritual. Most players can complete it in five minutes once they understand the flow.

### Step 1: Record the Server Seed Hash

Before your first bet, open Provably Fair or Settings on the platform. Copy the displayed **server seed hash**. Save it in notes or a screenshot. That is your pre-game lock.

### Step 2: Play and Record Results

Play normally. Note:

- The active client seed.
- The nonce for the round you want to check (usually visible in bet history).
- The on-screen result (multiplier, roll value, mine layout, slot landed, etc.).

### Step 3: Rotate the Server Seed

After enough rounds, rotate the server seed (or wait for automatic rotation). The plaintext server seed is now revealed — the value that was previously only represented by hash.

### Step 4: Confirm the Hash Matches

Re-hash the revealed server seed with the same algorithm (SHA-256 on many sites). Compare to the hash you saved in Step 1. **They must match exactly.** If they do not, you have serious evidence of inconsistency.

### Step 5: Recompute the Outcome

Enter server seed, client seed, and nonce into the platform's **verification calculator** — or an open-source script from the operator's documentation or community repos. The tool runs `derive_game_result` and shows the expected outcome.

Compare with what appeared on screen when you played. **They must match exactly.**
